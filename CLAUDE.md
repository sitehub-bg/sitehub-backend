# CLAUDE.md — sitehub-backend

Project context and conventions for future Claude Code sessions. Read this before suggesting architectural changes — many decisions here were made deliberately after weighing alternatives.

## What this is

Multi-tenant backend for **sitehub.bg**. Phase 1 serves school websites; later expanding to other organization types. The product surface:

- **Public REST API** — consumed by school static websites and mobile app. Path-based client routing: `/web/v1/...` for SSG builds, `/mobile/v1/...` for mobile app. Truly public read endpoints, no user auth, tenant identified by subdomain.
- **Admin panel API** — used by school staff to edit their school's data. JWT-authenticated.
- **Auth API** — login / refresh / password reset; issues JWTs.

The static school websites are **not generated by this backend** — they fetch from the public API.

Context: this project is also a side project + interview portfolio piece for the author. Bias toward decisions that are defensible in a senior-engineer interview, but resist gold-plating.

## Architecture: Hexagonal (Ports & Adapters)

**Flat workspace layout** — every package directly under the workspace root, no `crates/` or `packages/` subfolder.

```
sitehub-bin ──► sitehub-public-api ─┐
            ├─► sitehub-admin-api  ─┤
            ├─► sitehub-auth-api   ─┼──► sitehub-app
            ├─► sitehub-storage    ─┤
            └─► sitehub-tokens     ─┘
```

### Packages

| Package | Role | Internal deps |
|---|---|---|
| `sitehub-app` | Domain, ports (traits), use cases, `AppError`. **Zero I/O.** | none |
| `sitehub-storage` | Driven adapter: SurrealDB repository implementations. | `sitehub-app` |
| `sitehub-tokens` | Driven adapter: token issuance & verification (currently JWT). | `sitehub-app` |
| `sitehub-public-api` | Driving adapter: REST for static sites and mobile app. Path-based routing: `/web/v1/...` and `/mobile/v1/...`. | `sitehub-app` |
| `sitehub-admin-api` | Driving adapter: REST for school staff. | `sitehub-app` |
| `sitehub-auth-api` | Driving adapter: login / tokens. | `sitehub-app` |
| `sitehub-bin` | Composition root, single binary. Wires concrete adapters into ports. | all of the above |

### Dependency rules (enforced by the package graph)

- **`sitehub-app` must never gain a dependency on a framework or I/O library.** No `axum`, no `surrealdb`, no `tokio` (other than what `async-trait` needs — `async-trait` itself is fine, it's a macro). If you find yourself wanting one, the design is wrong.
- **No adapter depends on another adapter.** Driving adapters and driven adapters share only `sitehub-app`.
- **`sitehub-bin` is the only place** that imports both adapter sides; it's the composition root.

### Why this layout (and not others)

- **3 packages (merge `bin` into `api`)**: rejected — user explicitly wanted real dependency isolation and faster incremental compiles.
- **Strict DDD 5+ packages (separate `domain` + `application`)**: rejected — domain types and use cases live together in `sitehub-app`. Split later only if business logic genuinely outgrows it.
- **One driving-adapter package with multiple route modules**: rejected — user wants each adapter (public/admin/auth) isolated so its deps don't recompile the others. Mobile API merged into public-api as path-based routing (`/mobile/v1/...`) since it reads the same data.

## Conventions

### Cargo / Rust

- **Edition 2024**, resolver **3**.
- All shared external dep versions declared in `[workspace.dependencies]` at the root `Cargo.toml`. Packages reference with `{ workspace = true }` and add features per-package.
- Internal packages also declared in `[workspace.dependencies]` so `sitehub-app = { workspace = true }` works everywhere.
- Release profile: `opt-level = "z"`, `lto = "fat"`, `codegen-units = 1`, `strip = "debuginfo"`, `overflow-checks = true`.

### Domain modeling (inside `sitehub-app`)

- **Newtype IDs** — `UserId(Uuid)`, `SchoolId(Uuid)`, never raw `Uuid`. Prevents passing the wrong ID type.
- **Value objects with private fields and parse-only constructors** — e.g., `Email::parse(&str) -> Result<Email, _>`. Once you hold an `Email`, it's valid; no re-validation needed.
- Module layout inside `sitehub-app/src/`: `domain/`, `ports/`, `use_cases/`, `error.rs`, `lib.rs`.

### Async traits

- Use `#[async_trait]` for trait definitions when they need `dyn Trait` (which they do, for `Arc<dyn Port>` wiring). Native AFIT isn't dyn-friendly enough on stable yet.

## Error handling

**Decided pattern** after the user tried per-layer typed errors and found the chains untrackable.

- **One `AppError` enum** in `sitehub-app/src/error.rs` (using `thiserror`), with ~5–7 semantic variants (`NotFound`, `Conflict`, `Validation`, `Unauthorized`, `Forbidden`, …) plus `Internal(#[from] anyhow::Error)`.
- **Adapters use `anyhow::Context`** internally to attach context at each step. `?` auto-converts to `AppError::Internal` via the `From<anyhow::Error>` impl, preserving the full chain.
- **One mapping point**: `IntoResponse for AppError` in each driving adapter (or a shared helper) maps variants to HTTP status codes. `Internal` logs the full chain server-side via `tracing::error!`, returns a generic message to clients.
- `AppResult<T>` = `Result<T, AppError>` is the standard return type across `sitehub-app`.

**Do not:**
- Create a separate `errors` package.
- Add per-package error enums that wrap each other in chains.
- Hide the anyhow context by converting too early.

## Multi-tenancy

- **Expected scale**: ~5000 tenants (schools, then other org types).
- **Tenant identification**: by request **subdomain** (`acme.sitehub.bg`). Admin/mobile APIs identify tenant from a JWT claim instead.
- **Storage strategy**: **single SurrealDB database** with `tenant_id` on every record. Rejected DB-per-tenant after weighing trade-offs — migration burden at 5000 DBs is the killer (every schema change runs 5000 times with failure modes).
- **Defense-in-depth (planned)**: SurrealDB `DEFINE ACCESS` rules so the database itself enforces tenant isolation. Even if app code forgets `WHERE tenant_id = ?`, the DB rejects the query.
- **Compile-time enforcement**: every repository port method takes a `TenantCtx` parameter. No use case can forget to scope a query — the function signature requires the context.

## Authentication / Authorization

- **JWT** based.
- **Identity model**: `User` is the human, `Membership(user, school, role)` is the relation. Today every admin has at most one membership ("1 school per admin"), but the schema natively supports multi-school later. **Do not** put `school_id` directly on `User`.
- **Token issuance** goes through the app: `auth-api` calls a use case that uses a `TokenIssuer` port. Concrete impl in `sitehub-tokens`.
- **Token verification** is pure crypto (signature + expiry) — done in middleware in each driving adapter that needs it. **Does not** round-trip through `sitehub-app`. The verifier lives in `sitehub-tokens`.
- **Public API has no auth** beyond tenant identification via subdomain.

## Routing (one binary, multiple driving adapters)

`sitehub-bin` dispatches on `Host` header / subdomain:

| URL pattern | Adapter | Tenant from |
|---|---|---|
| `admin.sitehub.bg/...` | `admin-api` | JWT claim |
| `auth.sitehub.bg/...` | `auth-api` | login flow |
| `*.sitehub.bg/...` | `public-api` | subdomain |

Reserved subdomains (`admin`, `auth`, `api`) are not tenant-resolvable.

Reserved subdomains (`admin`, `auth`, `api`) are not tenant-resolvable. All other subdomains resolve to a school tenant.

Each driving adapter package exposes a `pub fn router(state: AppState) -> axum::Router`. `sitehub-bin` composes them.

## Open decisions (not yet made)

- **Static vs dynamic dispatch for ports**: leaning `Arc<dyn Port>` (dynamic) because cost is negligible for HTTP request paths and it's simpler. Not finalized.
- **Caching**: none yet. Re-evaluate after measuring with SurrealDB.
- **Password hashing**: probably `argon2` when we wire auth.
- **Migration tooling**: SurrealQL `DEFINE` statements in versioned files; runner not chosen.
- **Domain entity catalog**: school-specific entities (pages, news, staff, schedules, events) to be designed as features are built.

## What NOT to do

These are settled decisions — do not re-litigate without explicit user request:

- Don't add I/O or framework dependencies to `sitehub-app`.
- Don't create an `errors` package or per-package error enums.
- Don't merge `sitehub-bin` into any other package.
- Don't put `school_id` directly on `User` — use the membership relation.
- Don't bypass `TenantCtx` in repository methods.
- Don't switch to DB-per-tenant without re-analyzing the migration cost at 5000 tenants.
- Don't introduce a shared "common utilities" package without a concrete cross-cutting concern.
- Don't use the `crates/` or `packages/` subfolder — layout is flat.
