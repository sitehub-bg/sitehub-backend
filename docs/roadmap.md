# sitehub — Roadmap

Date: 2026-05-25
Principle: AI in the DNA. Every phase uses AI to build, operate, and scale.

## Phase 0: Foundation (Weeks 1-4)
*Goal: Backend can store and serve school content. One school works end-to-end.*

### 0.1 Domain model (sitehub-app)
- [ ] School entity (name, slug, domain, features, theme, locale)
- [ ] TenantCtx with compile-time enforcement
- [ ] Content types: Page, NewsPost, StaffMember, Document, GalleryImage
- [ ] Newtype IDs: SchoolId, PageId, NewsPostId, etc.
- [ ] Value objects: Email, Slug, Locale
- [ ] Feature registry enum (About, News, Staff, Admissions, etc.)
- [ ] AppError with semantic variants
- [ ] Port traits: SchoolRepository, ContentRepository, MediaStorage

**AI role:** Claude builds domain model from discovery document. Every entity traces back to the 103-school research.

### 0.2 Storage adapter (sitehub-storage)
- [ ] SurrealDB connection + tenant-scoped queries
- [ ] Repository implementations for all ports
- [ ] Schema definitions (DEFINE TABLE, DEFINE FIELD)
- [ ] Seed data: МГ Гео Милев with real content structure

**AI role:** Claude generates SurrealQL schema from domain model. Tests generated with AI.

### 0.3 Public API (sitehub-public-api)
- [ ] GET /schools/:slug — school config + enabled features
- [ ] GET /schools/:slug/pages — all published pages
- [ ] GET /schools/:slug/news — news feed (paginated)
- [ ] GET /schools/:slug/staff — staff directory
- [ ] GET /schools/:slug/documents — document listing
- [ ] GET /schools/:slug/gallery — gallery images
- [ ] Subdomain-based tenant resolution middleware

**AI role:** Claude generates API handlers, tests, and OpenAPI spec.

### 0.4 Auth API (sitehub-auth-api)
- [ ] POST /login — email + password → JWT
- [ ] POST /refresh — refresh token
- [ ] JWT claims: user_id, school_id, role
- [ ] Password hashing (argon2)

**AI role:** Standard auth flow, Claude implements end-to-end.

### 0.5 Admin API (sitehub-admin-api)
- [ ] CRUD for all content types (pages, news, staff, documents, gallery)
- [ ] Draft → Review → Publish workflow
- [ ] Media upload endpoint → Cloudflare R2
- [ ] School settings (features, theme, domain, locale)
- [ ] JWT-authenticated, tenant-scoped

**AI role:** Claude generates CRUD endpoints from domain model. Consistent patterns across all content types.

## Phase 2: Product-Market Fit (Weeks 9-16)
*Goal: 5-10 schools onboarded. Prove the model works.*

### 2.1 Self-service onboarding
- [ ] School signup flow (create account, pick slug, choose sections)
- [ ] Auto-generate Astro project from template
- [ ] Auto-deploy to Cloudflare Pages
- [ ] Welcome email with admin panel link

**AI role:** AI generates per-school Astro project from config. Zero manual setup.

### 2.2 Migration tool (generalized)
- [ ] Generic school website scraper (handles WordPress, Joomla, static PHP)
- [ ] Content classification: AI identifies which scraped pages are news vs about vs staff
- [ ] Image extraction and upload to R2
- [ ] Migration report: what was imported, what needs manual review

**AI role:** This IS an AI feature. LLM classifies scraped content into sitehub content types. Extracts structured data (staff names, roles) from unstructured HTML.

### 2.3 Content assistance (AI feature in the product)
- [ ] AI-powered content suggestions: "Your About page is missing Mission and Vision"
- [ ] Auto-generate meta descriptions for SEO
- [ ] AI translation: one-click translate BG content to EN (for multi-language sites)
- [ ] Content quality check: spelling, broken links, missing alt text

**AI role:** This is AI as a PRODUCT FEATURE, not just a build tool. Schools get AI-powered content assistance in the admin panel.

### 2.4 Compliance checker (AI feature)
- [ ] Scan school site against legally required sections
- [ ] Report: "Missing: Budget transparency (required by law), Data protection policy (GDPR)"
- [ ] One-click scaffold: generate template pages for missing required sections
- [ ] Accessibility audit: automated WCAG 2.1 AA check

**AI role:** AI audits compliance. Generates template content for legally required sections. This is a selling point.

### 2.5 Onboard 5-10 Pleven schools
- [ ] Migrate content from existing sites
- [ ] Train school admins (Saturday sessions)
- [ ] Collect feedback, iterate

## AI in the DNA — Summary

| Layer | How AI is used |
|---|---|
| **Build** | Claude writes the code, generates tests, creates schemas, builds UI components |
| **Migrate** | AI scrapes old sites, classifies content, extracts structured data from HTML |
| **Product** | AI content assistance, compliance checking, translation, quality audits in the admin panel |
| **Operate** | AI chatbot for first-line support, auto-generated FAQ, ticket classification |
| **Scale** | AI auto-configures new schools, generates starter content, suggests section layouts by school type |
| **Growth** | AI generates reports, writes news from photos, prompts seasonal content updates |

The product isn't just built WITH AI — it uses AI as a core feature that schools interact with daily. That's what "AI in the DNA" means.
